Keep client session alive

This request resets the timeout which destroys the client session (created with auth request) after 60s without any request. The session will be destroyed if not used for 60s (no requests, no websocket connection).

If there is an active websocket connection, it is not neccessary to use this request. The session will be kept alive as long as the websocket connection is active.

Note that this session has nothing todo with the opcua session. Its the general client session which may contain multiple opcua sessions. The opcua session has its own timeout which can be configured with the timeout parameter in the opcua session request.

Create a new session

Creates a new opcua session, establish a connection to OPC UA server.

Calling this function successfully creates a secure channel to the server endpoint and it creates and activates an application session in the server application. If the connection was lost the session will be reconnected. If the server was restarted the session will be recreated.

Without any request, the session is automatically disconnect and deleted after a configurable timeout period. This prevents any resource leaks on the server side where dead sessions are kept alive forever. The timeout time can be configured in milliseconds with the timeout parameter. The default value is 30 seconds and with zero the timeout is disabled. To reset the timeout cyclically, use the HEAD request. An interval of timeout/2 is recommended.

monitoringMode
string (MonitoringMode)
Default: "reporting"
Enum: "disabled" "sampling" "reporting"

The monitoringMode is an enumeration that specifies whether sampling and reporting are enabled or disabled for a MonitoredItem. This paramter is helpful to reduce performance overhead for not used monitored items but still keep them for next use. When a MonitoredItem is enabled (i.e. when the monitoringMode is changed from 'disabled' to 'sampling' or 'reporting'), the Server shall report the first sample as soon as possible and the time of this sample becomes the starting point for the next sampling interval. Setting the mode to 'disabled' causes all queued notifications to be deleted.

disabled - The item being monitored is not sampled or evaluated, and Notifications are not generated or queued. Notification reporting is disabled.
sampling - The item being monitored is sampled and evaluated, and Notifications are generated and queued. Notification reporting is disabled.
reporting - (default) The item being monitored is sampled and evaluated, and Notifications are generated and queued. Notification reporting is enabled.

Combine multiple requests with same starting path

Combine multiple requests with same starting path segments by using JSON batching.
The main goal is to reduce number of requests to the webserver and opc ua server. As performance optimization requests may be combined into one service call to opcua server if possible (i.e multiple GET value requests always handled as one). The order in which the requests are handled is not defined!
